3 problems with relational DBs


  • Scalability Problem with Relational databases: constantly having capacity problems as they got more and more data

  • Serialization/Deserialization: Translating data from database into objects that our application code can use requires significant design/coding work (vs. JSON)

  • Flexibility: The relational model is inflexible especially when adding a new field or data rules change

noSQL Solution


  • Vocabulary change: Relational databases store data in tables, whereas on noSQL databases like MongoDB, it stores data in collections.

  • Vocabulary change: The noSQL equivalent of a relational database record is called a document which are key-value pairs of data

  • In a collection, not every document has to look the same. Unlike traditional db where all records have to follow same structure

Schema-less


  • Schema-less: Collections do not need to be defined before they can be used, 2 documents in the same collection can have a different number of fields is inserted into it

  • A collection can be automatically created when the first document

  • Each document, can have a different format that evolves over time without requiring to modify document structure first

master-child relationship


  • Data relations such as master-child data can now be nested, eliminating need for table relationships, foreign keys and joins

  • Results in high performance and scalability

  • Suited to cloud hosting solutions which can automatically scale on demand

How does our node app talk to mongodb?

Mongoose

  • it's an object-document mapper (ODM) that allows a node application to interface with a MongoDB database

  • is the node equivalent of Entity Framework (ORM) when connecting .NET and SQL SVR. Install via npm.

  • provides a straight-forward, schema-based solution to model your application data. It includes built-in type casting, validation, query building, business logic hooks

Purpose of Mongoose Schema/Model

  • To map a specific collection in a MongoDB for a Node application

  • To set out validation rules for a MongoDB collection

  • To allow a model class to use the built-in mongoose CRUD methods



we create the models ourselves; these schemas live in the web app, not the db
